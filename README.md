# Описание
Утилита **docgen** - это инструмент, позволяющий в автономном режиме поддерживать актуальное состояние документации с помощью собственной агентной системы. Инструмент имеет прямой доступ к проекту через git-репозиторий, в котором содержится исходный код приложения (и не только), на основе чего и поддерживается документация.  
Для корректной работы **необходимо** желательно, чтобы анализируемый репозиторий уже содержал в себе хоть какую-то документацию для первоначального анализа.  
# Доступные команды
- **docgen init [OPTIONS]**  
Инициализация проекта, генерация **.docgen.yaml** и **.release-map.yaml**. Следует вызывать в отдельно созданной папке (не в папке репозитория). Здесь будет храниться вся документация по отдельным тегам, логи агента и **SUMMARY.md**, генерируемый им в ходе анализа.  
Опции:  
    * [required] --repo TEXT - URL git-репозитория;  
    * --github-token-env TEXT - Имя переменной окружения с GitHub-токеном (для доступа к API GitHub /releases/latest);  
    * --api-key TEXT - API-ключ LLM или имя переменной окружения, содержащей его;  
    * --base-url TEXT - Base URL для OpenAI-совместимого API;  
    * --model TEXT - Используемая модель.

- **docgen snapshot [OPTIONS]**  
Создаёт доверительную версию документации на основе всех md-файлов в репозитории, используя тег релиза. Папка снэпшота создаётся по имени тега.  
Опции:  
    * -r, --release TEXT - Тег релиза (v1.0, release-2024-01). По умолчанию — последний доступный релиз;  
    * -c, --check - режим полного аудита для проверки актуальности документации;  
    * -i, --iterations INTEGER - ручное указание максимального количества ходов при вызове инструментов;  
    * -l, --log - логирование действий;  
    * -v, --verbose - подробный вывод в консоли.

- **docgen watch [OPTIONS]**  
Запуск инструмента в режиме демона, в котором приложение через указанный промежуток времени будет проверять репозиторий на наличие новых тегов и поддерживать документацию в актуальном состоянии.  
Опции:  
    * -t, --interval INTEGER - интервал проверки новых релизов;  
    * -i, --iterations INTEGER - ручное указание максимального количества ходов при вызове инструментов;  
    * -l, --log - логирование действий;  
    * -v, --verbose - подробный вывод в консоли.

- **docgen versions**  
Просмотр имеющихся версий документации проекта. Актуальная документация, прошедшая ревизию агента, отмечена звездочкой.

# Использование
## Установка

Клонируем репозиторий и переходим в него:
```bash
git clone https://github.com/daisccho/docgen.git
cd docgen
```
Создаём виртуальное окружение и активируем его:
```bash
python3 -m venv .venv
source .venv/bin/activate
```
Обновляем pip и устанавливаем пакет в режиме разработки:
```bash
pip install --upgrade pip
pip install -e .
```
> [!NOTE]
> `pip install -e .` устанавливает команду `docgen` глобально — можно вызывать из любого места.
> Обновиться до новой версии можно комбинацией команд `git pull` и `pip install -e .`.

## CLI
Отдельно создаем папку для анализируемого проекта и инициализируем docgen:
```bash
mkdir project
docgen init --repo https://github.com/user/example.git
```
В папке появятся два файла: .docgen.yaml и .release-map.yaml. Залезаем в .docgen.yaml и меняем его на свой вкус. Структуру можно посмотреть [ниже](#docgenyaml).  

Далее необходимо создать снапшот документации изучаемого репозитория:
```bash
docgen snapshot
```
Без указания параметра -r создаётся документация на последнюю версию. С указанием **-r \<tag\>** будет произведен поиск по релиз-тегам и создана документация на указанный тег.
> [!IMPORTANT]
> Если в репозитории не будет найдено релиз-тегов, команда завершится ошибкой.  

При выполнении команды создаётся отдельная папка, названная по анализируемому тегу. Внутри данной папки будут находится все .md-документы, которые и представляют документацию по проекту. Агент также дополнительно сгенерирует SUMMARY.md, в котором опишет сам проект, его структуру и другую ключевую информацию.

Дальнейшая работа завязана на наблюдении за репозиторием:
```bash
docgen watch -t <time>
```
В **\<time\>** указывается промежуток, через который утилита опрашивает репозиторий на наличие новых релизов. При нахождении новых релизов: **(а)** создаётся новая директория на найденный релиз-тег, куда скачиваются вся существующая документация в данном релизе; **(б)** агентом производится проверка и сравнивание всех полученных файлов, и, на основе анализа кода и документации, он может их либо поменять, либо оставить как есть; **(в)** генерируется (либо дополняется старый) **CHANGELOG.md**, где перечисляются все изменения между версиями.

## Web-версия

Веб версия предоставляет все тот же функционал, но в виде веб-панели.  

Для начала нужно подготовить и запустить сервер веб-сайта:
```bash
python manage.py migrate
python manage.py createsuperuser
python manage.py runserver
```
В отдельном окне нужно также запустить worker'а:
```bash
python manage.py docgen_worker
```
Все остальные процедуры происходят непосредственно через браузер и не требуют разъяснений (я надеюсь).

# Инструменты ИИ-агента

- **terminal**  
Реализует функционал полноценного использования терминала. Имеет встроенный черный список, запрещяющий агенту выполнять особо опасные команды.

- **read_summary**  
Читает файл **SUMMARY.md**, содержащий в себе сводку по проекту. 

- **write_summary**  
Создаёт/изменяет **SUMMARY.md** по итогам работы агента с репозиторием.

- **collect_changelog**  
Собирает из репозитория файлы, содержащие описание изменений, дополняя их комментариями из коммитов.

# Файлы конфигурации

### .docgen.yaml
После запуска команды **docgen init** в cwd создается файл **.docgen.yaml**, который имеющий следующую структуру:
```yaml
config:
  git_repo: https://github.com/blank/repo # ссылка на репозиторий
  github_token_env: GITHUB_TOKEN  # переменная окружения с GitHub-токеном (опционально)
  llm_provider: openai # название провайдера, пока не используется
  llm_model: gpt-4o # название модели, указывать в соответствии с документацией провайдера
  llm_api_key: null  # или OPENAI_API_KEY в env
  llm_base_url: null # для OpenAI-совместимых провайдеров
  project_name: default  # имя проекта, пока не используется
  max_turns: 10 # максимальное количество шагов, доступное агенту на время исполнения задачи
```

### .release-map.yaml
Файл маппинга тегов релизов на хэши коммитов. Заполняется автоматически при создании снэпшотов и при работе watch. Редактирование вручную не требуется.
```yaml
last_documented_release: v1.0.0
releases:
  v0.1.0: a1b2c3d4e5f67890abcdef1234567890abcdef12
  v1.0.0: b2c3d4e5f67890abcdef1234567890abcdef1234
```

# Генерация снапшота

**Снапшот** - это сохраненная вручную документация, которая служит отправной точкой (в случае первого создания) для ИИ-агента в последующем поддержании документации проекта.  
Для генерации снапшота в папке проекта необходимо запустить следующую команду:
```bash
docgen snapshot
```
Без указания параметров снапшот будет сделан с последнего доступного релиза. В конфиге переменные **last_documented_release** и **releases** будут изменены, чтобы отразить текущее состояние проекта.  
При последующих вызовах **snapshot** будут генерироваться и другие версии документации, но в данном случае изменяться будет только **last_documented_release**. ИИ-агент при запуске в виде демона будет изменять документацию, ссылаясь на последний задукоментированный релиз.

# Автоматическое поддержание документации
При запущенном процессе с помощью **docgen watch** каждые **N** минут ИИ-агент будет генерировать новую документацию в отдельной папке, названной по имени тега релиза. Работа агента заключается в генерации в рабочей директории следующих файлов:
```yaml
docgen_project_dir:
    .release-map.yaml
    SUMMARY.md
    logs/:
        snapshot_20260708_135047.log
        watch_20260708_135137.log
    v0.2.0/:
        docs/:
            README.md
            Installation.md
            Webhooks.md
            ...
    v1.0.0/:   # условная стартовая документация, на основе неё 
        docs/:  # агент в дальнейшем генерирует все остальное
            README.md
            Installation.md
```
Артефакты после работы утилиты:
- **SUMMARY.md**  
Файл, генерируемый агентом, в котором содержится сводка по проекту, над которым он работает. Агент читает этот файл каждый раз, когда начинает процедуру обновления документации, чтобы поддерживать свои знания о проекте. Данный файл обновляется в конце рабочего цикла **watch** перед уходом в сон.

- **snapshot/<snapshot|watch>_date_time.log**  
Лог-файлы, в которых записываются действия агента от момента запуска команды до её завершения. Все логи храняться в директории **logs/** в корневой папке.

- Различные **.md**-файлы, в которых и содержится документация. Старая обновляется на основе новой информации, новая (не фигурировавшая нигде ранее) добавляется.

ИИ-агент в процессе своей работы самостоятельно просматривает исходную документацию, комментарии к коммитам, измененные файлы и т. д. и принимает решение, какие файлы необходимо изменить, какие добавить, какие убрать и прочее.
